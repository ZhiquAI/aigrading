## 💡 使用示例

### 1. 子代理驱动开发 (SDD)

最适合复杂功能的开发，会自动完成规划→实现→验证全流程：

```
用 SDD 方式为用户管理模块添加批量导入功能
```

**流程：**
1. 规划代理：分析需求，创建设计文档和实现计划
2. 实现代理：按照计划编写代码和测试
3. 审查代理：代码质量审查和验证
4. 验证代理：运行测试，确认功能完整

### 2. 系统化调试

当遇到难以定位的 bug 时：

```
调试用户登录后偶尔显示未登录的问题
```

**调试策略：**
- 根因追溯：从症状追踪到源头
- 压力测试：在不同条件下复现问题
- 防御深度：多层验证防止回归
- 条件等待：观察状态变化

### 3. 代码审查

完成功能开发后，请求代码审查：

```
帮我审查 components/Button.tsx 的代码
```

**审查维度：**
- 代码质量：类型安全、错误处理、DRY 原则
- 架构设计：可扩展性、性能、安全性
- 测试覆盖：边界情况、集成测试
- 生产就绪：向后兼容、文档完整性

### 4. 头脑风暴

探索功能需求和解决方案：

```
帮我头脑风暴如何改进用户注册流程的用户体验
```

### 5. TDD 开发

测试驱动开发：

```
用 TDD 方式为 MembershipCard 组件添加测试
```

## 🔄 工作流集成

### Subagent-Driven Development 工作流

```
需求分析 → 设计规划 → 代码实现 → 质量审查 → 功能验证
   ↓         ↓         ↓         ↓         ↓
brainstorming → writing-plans → executing-plans → requesting-code-review → verification
```

### 完整开发流程示例

```bash
# 1. 头脑风暴需求
"帮我头脑风暴用户权限管理功能"

# 2. 用 SDD 开发
"用 SDD 方式实现用户权限管理功能"
# 自动完成：设计文档 → 实现计划 → 代码编写 → 测试 → 审查

# 3. 完成分支
"我完成了用户权限功能，准备合并"
```

## 📋 审查报告格式

代码审查会生成结构化报告：

### 优点
- 具体说明做得好的地方

### 问题
- **Critical** (必须修复): Bug、安全问题、功能缺陷
- **Important** (应该修复): 架构问题、缺失功能
- **Minor** (建议改进): 代码风格、优化机会

### 评估
- **可以合并**: 是/否/修复后
- **理由**: 技术评估

## 🎓 最佳实践

1. **复杂功能使用 SDD**：自动完成完整开发流程
2. **定期代码审查**：每次任务完成后请求审查
3. **系统化调试**：不要盲目试错，使用结构化方法
4. **测试驱动**：关键功能使用 TDD 保证质量
